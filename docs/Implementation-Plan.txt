Here is the comprehensive, phase-wise implementation plan for the **NodeCode IDE**. This roadmap synthesizes all prior documentation (PRD, TRD, UI/UX Brief, App Flow, and Backend Schema) into an aggressive, structured **24-week development lifecycle**.

It is divided into 6 clear phases, organizing engineering tasks across frontend, backend, and DevOps domains.

---

## Phase 1: Foundation, Security & Database Infrastructure (Weeks 1–3)

**Objective:** Establish the core Next.js monorepo, set up the PostgreSQL database schema, and finalize the authentication flow so users can create isolated workspaces.

* **Reference Documents:** Backend Schema (Section 1 & 2), App Flow (Landing Page & Dashboard).
* **Key Engineering Tasks:**
* **DevOps:** Initialize the Next.js (App Router) project with TailwindCSS and Shadcn/ui. Set up the local Docker environment for the PostgreSQL database.
* **Backend (Schema):** Configure Prisma ORM and migrate the 5 core tables (`User`, `Workspace`, `CanvasState`, `CodeModule`, `CodeEntity`). Implement the Generalized Inverted Index (GIN) on the `CanvasState.nodes` column.
* **Authentication (Flow):** Implement the GitHub OAuth 2.0 flow. Create the `/api/auth/callback` route, handle the user profile upsert, and establish the stateless JWT cookie session.
* **Frontend (UI/UX):** Build the Dashboard Hub UI using the "Abyss Black" and "Matte Graphite" color palette. Create the repository import modal.


* **Phase 1 Milestone:** A user can log in via GitHub, view a dark-themed dashboard, and successfully create a new "Workspace" entry in the database.

---

## Phase 2: The Ingestion Pipeline & Visual Canvas (Weeks 4–8)

**Objective:** Build the core Abstract Syntax Tree (AST) parser capable of reading TypeScript files and rendering them instantly onto the interactive React Flow canvas.

* **Reference Documents:** TRD (Section 3), PRD (Flow A), UI/UX (The Visual IDE Studio).
* **Key Engineering Tasks:**
* **Backend (AST Engine):** Integrate `ts-morph` and `@octokit/rest`. Write the `/api/repo/ingest` endpoint to clone a target GitHub repo into a temporary volume, parse the `.ts` files, and output the structured JSON.
* **Frontend (State):** Implement the Zustand global store to handle the `CanvasState` JSON without triggering unnecessary DOM re-renders.
* **Frontend (Canvas UI):** Integrate `@xyflow/react`. Design the custom "Module Node" and "Function Node" components with the "Electric Blue" input/output ports and "JetBrains Mono" typography.
* **Animation (UI/UX):** Implement the active/static state bezier curves for the edges (data flow lines).


* **Phase 2 Milestone:** A user can paste a GitHub URL, the backend parses it, and the frontend beautifully renders a read-only, interactive node graph of the codebase.

---

## Phase 3: Bi-Directional Sync & The Split-Screen IDE (Weeks 9–12)

**Objective:** Bridge the visual graph and the underlying text code. Enable users to edit nodes on the canvas and have the backend safely rewrite the physical files.

* **Reference Documents:** TRD (Section 1 & 3), App Flow (Edge Cases).
* **Key Engineering Tasks:**
* **Frontend (Editor):** Integrate `@monaco-editor/react` into the right 30% panel of the IDE Studio. Apply the borderless, dark custom theme to match the canvas.
* **Backend (Reverse Engine):** Write the `/api/canvas/sync` endpoint. Program `ts-morph` to receive JSON patches, locate the exact file/function, modify the AST directly, and execute `.saveSync()`.
* **State Sync & Debouncing:** Implement a 300ms debounce on the frontend so dragging a node updates the UI instantly but waits to push the DB/File update to prevent network thrashing.
* **Error Handling (App Flow):** Implement the "Syntax Breakage" lock. If the Monaco editor detects invalid TS code, the canvas locks, the footer turns "Cyber Red," and graph modifications are disabled until the syntax is fixed.


* **Phase 3 Milestone:** The "Single Source of Truth" is achieved. Deleting a node deletes the code; writing a new function in the editor spawns a new node on the canvas.

---

## Phase 4: AI Scaffolding & Local Execution Runner (Weeks 13–17)

**Objective:** Implement the generative AI architecture to allow greenfield project scaffolding from natural language, and build the local CLI runner to execute the code.

* **Reference Documents:** PRD (Flow B), TRD (Section 6 & 7).
* **Key Engineering Tasks:**
* **Backend (AI Engine):** Integrate the Vercel AI SDK. Write the multi-step prompt orchestration: Step 1 forces the LLM to output only the structural `CanvasState` JSON. Step 2 spins up parallel sub-workers to write the inner TypeScript logic for each node.
* **Frontend (AI Overlay):** Build the "Matrix Green" animated loading overlay where users watch the structural graph generate node-by-node before approving the code infill.
* **DevOps (CLI Runner):** Develop the `nodecode-runner` npm package. Build the WebSocket connection that binds the browser session (using `WorkspaceToken`) to the local terminal.
* **Docker Integration:** Program the backend to automatically generate `docker-compose.yml` envelopes for AI-generated projects.


* **Phase 4 Milestone:** A user types "Build a TS Auth API", watches the graph draw itself, approves it, and runs `npm exec nodecode-runner` to start the working API on their local machine.

---

## Phase 5: The "Daily Driver" IDE Features (Weeks 18–21)

**Objective:** Transform the visualizer into a fully functional development environment by adding physical file navigation, local command execution, and version control, ensuring the developer never has to leave the browser.

* **Reference Documents:** TRD (Section 7 - Local Execution Runner), App Flow (The Visual IDE Studio), Backend Schema (`WorkspaceToken`).
* **Key Engineering Tasks:**

### 1. The Source Code Tree (File Explorer)

* **Backend Task:** Create a lightweight API route (`/api/workspace/tree`) that uses Node's native `fs/promises` to recursively map the cloned repository's directory structure (ignoring `node_modules` and `.git`).
* **Frontend Task:** Implement a React tree-view component in the left sidebar (15% width).
* **State Integration:** Bind the tree to the Zustand store. Clicking a `.ts` file in the tree should instantly filter the React Flow canvas to focus on the module node for that specific file.

### 2. Integrated Terminal & Quick Actions

* **Frontend Task:** Integrate `xterm.js` into a collapsible bottom drawer. Style it with the "Abyss Black" theme.
* **Backend / Local Runner Task:** Expand the `nodecode-runner` CLI tool. It must establish a bi-directional WebSocket connection (`socket.io` or native WebSockets) that pipes the standard input/output (stdin/stdout) of the local machine's shell directly into the browser's `xterm.js` instance.
* **Quick Actions UI:** Build a row of predefined action buttons above the terminal. When clicked, these send predefined strings via the WebSocket to execute locally:
  * `▶ Run Dev Server` (Executes `npm run dev`)
  * `📦 Install Dependencies` (Executes `npm install`)
  * `🧹 Lint & Format` (Executes `npm run lint`)

### 3. Version Control (Git UI)

* **Backend / Local Runner Task:** Write secure, sanitized wrapper functions around the local machine's Git CLI (e.g., `exec('git status')`).
* **Frontend Task:** Add a "Source Control" tab next to the File Tree in the left sidebar.
* **Diff & Commit UI:**
  * Fetch the output of `git status` and list modified files in the sidebar.
  * Clicking a modified file opens the **Monaco Diff Editor** (a built-in feature of Monaco) in the right-hand panel, showing the before-and-after of the AI or manual code changes.
  * Add a text input for the commit message and a primary "Commit & Push" button that executes `git add .`, `git commit -m "[message]"`, and `git push`.


* **Phase 5 Milestone:** The user can navigate their physical files via the left sidebar, spin up their local server using a quick-action button in the integrated terminal, and commit their AI-generated code directly to GitHub using the built-in diff viewer.

## *Updated* Phase 6: Optimization, Testing & Beta Launch (Weeks 22–24)

*(Shifted back by 3 weeks to accommodate Phase 5)*

**Objective:** System hardening, performance tuning, and rollout.

* **Reference Documents:** PRD (Success Metrics), UI/UX (Motion & Polish).
* **Key Engineering Tasks:**
* **Performance:** Implement React Flow viewport virtualization so the browser can handle up to 2,000 nodes at 60 FPS without crashing.
* **Infrastructure:** Finalize AWS ECS Fargate deployment for heavy AST parsing.
* **Security Audit:** Ensure the terminal WebSocket connection strictly validates the `WorkspaceToken` so unauthorized users cannot execute arbitrary commands on a user's local machine.


* **Phase 6 Milestone:** Platform is stable. Time to First Graph (TTFG) is under 10 seconds, and the AST Parse Error rate is below 5%. Ready for developer onboarding.
